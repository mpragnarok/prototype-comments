§0 已鎖定的設計決策

使用者 2026-05-30 核准，不再反覆。以下四項決策為本系統設計的基石。
# 決策 內容
D1 回覆階層 = 單層 採 Figma/Linear/Notion 模式：root 留言可「回覆 + resolve/unresolve」；replies（第二層）不可再回覆、不可 resolve。不做巢狀第三層。
D2 Emoji = 精選 6–8 顆 不掛整套 iPhone emoji（依賴大、CP 值低）。reactions 定位為快速情緒，非完整表情輸入。
D3 留言 bar 浮動底部 bar 固定浮在畫面底部（非塞進 header）。連同主色 teal #0FA0A0、只有 pin 用系統紅 #BA1A1A 寫成 design 規格 + 消費端專案規範不可變條款，每次開發都遵守。
D4 Pin 重設計 提 3 案（pill+編號 / teardrop 水滴 / 對話泡）給使用者選後實作。

色彩 / 行為不可變條款

主色 primary
teal #0FA0A0 — 所有互動 accent（focus、active chip、hover、mention）一律用此色。
系統紅 — 只給 pin
#BA1A1A 不得用於一般按鈕 / chip / 連結。
留言 bar 位置
永遠浮動在視窗底部，不注入頁面 header。
例外：reaction mine 狀態
#FFDAD6 / #BA1A1A 允許作為「你已按」語意，屬設計例外。

§3 Roadmap 評估

R1 ui-flow 多模式產出
現況：prototype-first — 設計師做好 prototype → 部署 Netlify → ui-flow skill 截圖 + overlay。
新模式：spec + Storybook → 直接生「設計稿 + ui-flow」，讓 flow 在 prototype 之前就存在。
  • 建議分階段：先做 R3 → 再由 stories 組合畫面 → 再由 spec 驅動 flow
  • PoC 對象：羽球比賽系統（已有 Storybook）
效益高 / 成本高   可行性中
R2 設計稿留言 → 自動 report.html
目標：每畫面 = 截圖 + unresolved 留言（標記點 + 列表）。雙用途：人閱讀 + AI 修正設計稿的資料來源。
  • 管線：query Firestore → 截圖 → 合成 → 輸出 report.html + machine-readable JSON
  • 截圖管線已存在（prototype-flow-doc skill capture）
效益最高 / 成本中   可行性高
R3 UI component 畫面綁定 Storybook id
目標：ui-flow 畫面區塊對應到 Storybook component ID，建立 設計 ↔ component ↔ code 追溯。
  • 靠 data 屬性 / mapping manifest：data-storybook-id
  • 建議先做手動 manifest，再自動化
  • R1 的前置基石
效益中 / 成本中高   可行性中高
R4 dev note 可讀性
目標：改善 ui-flow skill 產 dev note 那段的輸出品質 — 白話、行動導向、依 tag 分組、去術語。
  • 檢視：~/.claude/skills/prototype-flow-doc/scripts/03-generate*
  • 連同 C3 一起做
效益中 / 成本低   快速 win

§4.6 pc.js 發布策略

痛點：pc.js 是 build.py 產的 bundle，各消費端各存一份 copy；改一次要 cp + 重部署多處。
技術前提：消費端是靜態 HTML 設計站（無 bundler / npm build step），用 ESM import { initPrototypeComments } 引入。
拍板順序（使用者同意，現在不過早正式化）：
  1. 先把手上 C2 / A3 / B5 收尾（已完成）
  2. 執行 Option A：通用 kit 全收進 prototype-comments（含 skill source 放 repo/skills/ + symlink 到 ~/.claude/skills/，讓 skill 跟 kit 一起版控）
  3. 用一陣子，等出現「要 publish / 要 semver / 多個 consumer」其中之一，才考慮正式 package 化
A. jsDelivr from GitHub（傾向）
build → commit dist/pc.js → git tag → 各 HTML 改引 cdn.jsdelivr.net/gh/...。零複製、版本可控、免 npm publish。@<tag> 可 pin；@latest 自動更新但有 CDN 快取延遲。
B. npm publish + CDN
正式 @scope/pc，npm i（給有 bundler 的 consumer）+ unpkg/jsDelivr（給靜態站）。版本生態完整，但要 npm 帳號 + publish 流程。
C. 維持複製但自動化
build.py 一次輸出到所有消費端路徑 + sync script。治標，仍是多份 copy。
