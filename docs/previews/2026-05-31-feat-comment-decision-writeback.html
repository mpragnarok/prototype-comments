留言決議寫回：採用 / 不採用 / 待議 + 註記

設計稿留言報告升級成「可決策」：每則 root 留言能標記 採用 / 不採用 / 待議 並附一句註記，按下即寫回 Firestore。pc.js overlay 與 report.html 共用同一個 decision 欄位，兩邊都看得到決議、AI 讀 report.json 就知道哪些要改。本檔是 GATE plan，核准前不動 code。

Why — 報告現在只能讀，不能決策

目前流程：設計師/工程在 pc.js 留言 → 05-comment-report.js 蒐集成 report.html/json。報告是唯讀的：人看完留言後，「要不要採用」這個決定散落在口頭、聊天、記憶裡，沒有落到資料上。AI 拿到 report.json 也只看到留言內容，不知道哪些拍板要改、哪些被否決。

  • 沒有「決議」這個 first-class 狀態 → 同一則留言反覆被討論
  • resolved 只表示「thread 處理完」，不等於「設計採不採用」（兩者正交）
  • AI 修設計稿時無法分辨「這留言要照做」vs「這留言已被否決」

方案 / 理由
決議寫在哪
arch
兩邊都可寫（A3）：report.html 是主決策面（人讀完一次拍板），pc.js overlay 也能設（看原型當下順手標）。兩邊共用同一 decision 欄位、同一套寫入邏輯語意。符合你說的「人直接在報告下決定」。
decision vs resolved
arch
正交、互不連動。decision=設計拍板；resolved=thread 收尾。可「採用但未 resolve」。不自動連動（避免隱性副作用）。
不採用的顏色
不可變
用灰 #6b7280，不可用紅。design-spec 不可變條款「紅只給 pin / @mention」。不採用＝低調灰（與 resolved pin 同色系），採用＝olive，待議＝clay。
誰能改決議
rules
任何登入者（reviewer 通常不是留言作者）。靠 rules 限定「只能改 decision/decisionNote 兩欄」，不能改 body/author。
report.html 變重
ux
report.html 需內嵌 Firebase SDK + Google 登入才能寫回。未登入時三鍵 disabled、僅顯示既有決議（唯讀），不破壞「純讀報告」的輕量情境。

Firestore rules 變更（精確 diff）

現況（design-spec L155 記載）：update 允許「作者本人，或只改 ['resolved','reactions']」。要讓非作者 reviewer 寫決議，把白名單兩欄加進去：

firestore.rules · match /prototype-comments/{project}/comments/{id}
// 允許：作者本人改全部，或任何登入者只改白名單欄位
allow update: if request.auth != null && (
  resource.data.authorUid == request.auth.uid ||
  request.resource.data.diff(resource.data).affectedKeys()
    .hasOnly(['resolved', 'reactions', 'decision', 'decisionNote'])
);

驗收：deploy 後用「非作者帳號設 decision」應成功；用同帳號試改 body 應被拒（PERMISSION_DENIED）。需先確認各 Firebase 專案（jubo-badminton-design / prototype-comments-27106）目前的 rules 實體與此一致。

給你確認的問題

✅ Q1 已定 · 決議只貼「主留言」
「採用／不採用／待議」只貼在每串最上面的主留言（＝一個提案）；它的回覆不單獨給決議（回覆是補充討論，不是獨立提案）。positional pin 與 note 的 root 都適用。（使用者表示原問題聽不懂，於 goal 模式採此最單純預設定案，可反悔）
✅ Q2 已定 · 採用不自動 resolve
留言有兩個獨立狀態：resolved（這串收掉了）與新的 decision（採用／不採用）。按「採用」不會自動把該串標成已解決——兩者正交（可能「採用了但討論還沒完」）。最單純、無隱性副作用。（同上，goal 模式定案，可反悔）
✅ Q3 已定 · report.html 寫回需 Google 登入
Firestore 寫入一定要 auth。report.html 內嵌 Google 登入；未登入＝唯讀看決議。使用者已答 OK。
✅ 基礎定案 · 不採用用灰不用紅、decision 與 resolved 正交、純 additive 無 migration
依不可變條款與現有資料模型直接定案。
