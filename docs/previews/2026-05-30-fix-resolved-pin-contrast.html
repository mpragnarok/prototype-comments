Resolved pin 對比度修正 ＋ 開發流程定案

緣由 使用者回報：已解決（resolved）的 pin「有點看不清楚」。根因＝淡灰底 #d1d5db + 灰字 #6b7280 + opacity .7 三層疊加，對比度僅 ~2.5:1，遠低於 WCAG AA 的 4.5:1。本計畫同時把「改規格前先 plan、實作後更新回來源文件」的開發流程固化。

01

問題：現況 vs 未解決

背景刻意用淺色設計站底色 #eef5f4（pin 實際所在環境）。現況 resolved pin 的 ✓1 幾乎糊進背景。

02

方案比較

三個候選都把「灰字」改成「白字 + 較深的灰底」，並把 opacity 從 .7 提高，確保可讀。維持「已解決＝低調、退到背景」的語意，不違反「紅只給未解決 pin」的不可變條款。

建議 A。 清晰達 AA，又保留「次要」語意。C 雖最清楚，但白描邊讓 resolved pin 變顯眼，跟它該有的低調定位衝突；B 在淺底上略飄、未完全達 AA。RECOMMEND A
03

核准後要動的檔案

採方案 A 時，下列三處一起改（程式碼 + 來源規格文件同步，不留落差）。

code
src/styles.js · .pc-pin.resolved → background:#6b7280; color:#fff; opacity:.80（含 ::before 尾巴同色）。改後 build.py rebuild + node --check dist/pc.js。
spec
設計規範來源 docs/design/prototype-comments-design-spec.html · L87 色票 swatch、L145 pin.resolved token：#d1d5db / opacity .6 → #6b7280 / #fff / opacity .80。
spec
SSOT roadmap docs/2026-05-30-comment-system-and-uiflow-roadmap.md · §2.A 加一筆 fix 記錄。
test
visual fixture 補「resolved vs 未解決 同 threadCount 並排」cell，--update baseline；e2e 斷言 resolve toggle 後 pin 有 resolved class。
04

開發流程定案（寫進 AGENTS.md）

本次起，任何「設計規範 / 規格」變更一律走這條鏈。AGENTS.md 與 CLAUDE.md 只 reference 來源文件（design-spec / roadmap / 各 spec），不再把設計內容內嵌進去。

1

提 plan（html-doc 格式）

改規格前，先用 html-doc 出一份「要怎麼改」的計畫（就是這份），含問題、方案、影響檔案。

2

使用者 review → 確認

等明確核准才動工。GATE · 必須卡關

3

實作

改 code（pc.js src）＋ 對應 test（功能必附 e2e；畫面改動必附 visual regression）。

4

更新回來源文件

把定案同步回 design-spec.html / roadmap / spec doc。這些是設計規範的單一來源（SSOT）。

5

AGENTS.md / CLAUDE.md 只放 reference

doctrine 檔案僅指向上述來源文件 + 本流程，不重複設計細節。要查設計規則 → 點連結到 design-spec。
